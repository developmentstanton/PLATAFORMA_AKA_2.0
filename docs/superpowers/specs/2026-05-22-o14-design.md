## 1. Contexto y propósito

O14 es un informe que hoy existe como reporte **Power BI embebido** en el portal viejo (`aka/`). plataforma_20 está reemplazando esos Power BI por informes **nativos** (PHP + SQL Server + ECharts/HTML, sin npm/build). G00 ya se construyó así; O14 sigue el mismo patrón.

O14 no es solo un dashboard de consulta: es una **herramienta operativa de rebalanceo de curva de tallas**. Cruza el inventario ideal (*siembra*) contra el disponible real y el que viene en camino (*hold*) para detectar, por talla, dónde **sobra** y dónde **falta**, y recomendar **reubicaciones entre tiendas**, **solicitudes a CEDI** y **solicitudes al proveedor**.

El informe es **B2B y multi-proveedor**: cada proveedor logueado ve solo sus referencias (igual que G00, `WHERE proveedor` implícito vía `$_SESSION['proveedor']`).

## 2. Glosario

| Término | Definición |
|---------|------------|
| **Negocio** | Una combinación **referencia + color** (un SKU a nivel ref-color, ej. `434500 / NEG`). NO incluye talla. |
| **Tienda** | Una bodega/punto de venta (`INTEGRACION.dbo.Bodegas`, cía 7). Se agrupa por **Grupo Bodegas** (AKA, ZEUS, …). |
| **Talla** | Talla del producto (ej. 7, 7.5, 8 … 11). **Dinámica** según el tipo de producto. |
| **Siembra** | Inventario **ideal** que debería tener cada tienda por negocio×talla. Sale de un SELECT a una tabla ya definida. |
| **Total Stock** | Inventario real actual en tienda. |
| **Disponible** | Medida operativa de inventario usada en la fórmula de balance (definición exacta vs Total Stock: **a confirmar con tablas**). |
| **Hold entrante** | Unidades en camino hacia la tienda (en tránsito de entrada). |
| **Ventas** | Unidades vendidas en el rango de fechas. Solo **informativa** (no ajusta la siembra). |
| **CEDI** | Centro de distribución / bodega central. Fuente intermedia de abastecimiento (identidad en datos: **a confirmar**). |
| **Sobrante / Faltante** | Derivadas: `Siembra − (Disponible + Hold)`. Resultado **positivo → Faltante**, **negativo → Sobrante**. |

## 4. Arquitectura y archivos

```

api/informe_o14.php       endpoint JSON: ?tab=b|c|reco &<filtros> (&ref=<ref>&color=<color> para C/reco)

cache/                    reusa #refs del proveedor + catálogos de filtros (TTL diario, patrón G00)
```

**Flujo de datos:** request → resolver proveedor (sesión) → `#refs` (referencias del proveedor, cacheadas) → query(s) **tidy** contra siembra / inventario / hold / ventas acotadas a esas refs → PHP arma `kpis` + `filas` tidy → JS pivota a la matriz ancha. El tab *Recomendaciones* pasa además los datos tidy por `o14_recomendador.php` antes de responder.

Decisión de render (aprobada): **tidy-data + pivote en cliente**. El SQL devuelve filas largas `(clave, talla, medida, valor)`; PHP/JS construyen la matriz como **tabla HTML vanilla** con headers fijos, totales y expandir/colapsar. Sin dependencias nuevas; maneja tallas dinámicas de forma natural; alimenta directo al motor (que también requiere datos tidy).

## 5. Contrato de datos

**Grano base:** `tienda × negocio(ref+color) × talla`. O14B agrega sobre tienda; O14C filtra a un negocio.

**Fuentes (confirmadas 2026-05-22).** Todas cruzan por `(CIA, BODEGA, REFERENCIA, COLOR, TALLA)`. `NEGOCIO = REF-COLOR`, `LLAVE = CIA-BODEGA`.

| Medida | Fuente | Campo / filtro |
|--------|--------|----------------|
| Siembra (ideal) | `stanton.dbo.t400_cm_existencia` + joins (t150_mc_bodegas, t121_mc_items_extensiones, t120_mc_items) | `f400_cant_nivel_min_1`; COLOR=ext1, TALLA=ext2; `(nivel_min_1>0 OR nivel_pedido>0) AND referencia<>'GIFTCARD'` (cross-DB vía conexión integracion) |
| Disponible (= Total Stock) | `INTEGRACION.dbo.inv_actual_PBI` | `CANTIDAD`; `cia<>'001' AND COLUMNA1 IN ('INV1430','INV1435','400')`. **Total Stock = Disponible** (misma medida). |
| Hold entrante | `INTEGRACION.dbo._hold_actual_PBI` | `CANTIDAD` a `BODEGA_ENT`; `cia<>'001'` |
| Ventas | `INTEGRACION.dbo.Ventas_Detal_PBI` / `_Acum_PBI` | por REF+COLOR+TALLA+BODEGA y rango de fechas (como G00); informativa |
| CEDI (stock) | `inv_actual_PBI` con `BODEGA='CEDI'` | 2 CEDIs por cía: **`002-CEDI`=Brahma Concept**, **`007-CEDI`=Cauchosol**. Excluir de la matriz de tiendas; usar como fuente de cascada. |
| Sobrante / Faltante | derivada | `Siembra − (Disponible + Hold)`; + = Faltante, − = Sobrante |

**Alcance por cía (confirmado):** O14 opera **separado por cía**. La reubicación solo mueve entre tiendas de la misma cía y cada cía usa su propio CEDI (`002→Brahma`, `007→Cauchosol`); las matrices no mezclan unidades de negocio. El endpoint llama al motor por `(cia, negocio)`. Usar `LLAVE (cia-bodega)` como id de tienda (mismo BODEGA se repite entre cías — gotcha de G00).

**Dimensiones:** `ITEMS` (atributos de referencia para filtros; `#refs` acota al proveedor), `Bodegas` (tienda → Grupo, Ciudad, Depto, Coordenadas; **cía 7**, ver gotcha fan-out de G00).

**Salida del endpoint (tidy):**

## 6. Medidas y fórmulas

- **Sobrante / Faltante** (por talla, por fila): `balance = Siembra − (Disponible + Hold)`.
  - `Faltante = max(0, balance)`
  - `Sobrante = max(0, −balance)`
- **Ventas**: informativa; rango de fechas configurable ("Rango Ventas"). Stock/Siembra/Hold son snapshots actuales.
- **KPIs** (10 tarjetas de cabecera, en orden): Negocios, Siembra, Tiendas con Siembra, Negocios con Siembra, Total Stock, Tiendas con Inv, Ventas, Tiendas con Venta, Sobrantes, Faltante. (`Siembra` = total de unidades ideales; `Negocios con Siembra` = ref-colores con siembra definida; `Tiendas con Siembra` = tiendas con siembra definida.)

## 10. Motor de recomendaciones

Opera **por negocio + talla**. Para el negocio seleccionado (o "todos los negocios con faltante"), corre la cascada:

1. Por tienda calcula `Faltante`/`Sobrante = Siembra − (Disponible + Hold)`.
2. **Reubicación** — empareja sobrante→faltante (mismo negocio+talla) con **política de origen configurable** (default: *mayor sobrante primero*; **tunable**). El sobrante está por definición sobre el ideal, así que mover nunca deja a la tienda origen bajo su siembra. Genera `{origen, destino, talla, uds}`.
3. **CEDI** — el faltante restante se cubre con stock de CEDI si hay → `{destino, talla, uds}`.
4. **Proveedor** — lo que ni reubicación ni CEDI cubren → solicitud agregada `{talla, uds}`.

**Módulo `api/o14_recomendador.php`:** función pura `recomendar($tiendas, $cedi, $policy)` → plan. Sin acceso a BD → **testeable con fixtures**.

**Salida del plan:**
```json
{
  "negocio": { "ref":"434500", "color":"NEG" },
  "reubicaciones":         [ {"origen":"300","destino":"245","talla":"9","uds":2} ],
  "solicitudes_cedi":      [ {"destino":"245","talla":"10","uds":1} ],
  "solicitudes_proveedor": [ {"talla":"11","uds":4} ],
  "resumen": { "faltante_total":N, "por_reubicacion":N, "por_cedi":N, "a_proveedor":N }
}
```

**UI tab Recomendaciones:** selector de negocio (o "todos con faltante"), 3 bloques (Reubicaciones origen→destino · Solicitar a CEDI · Solicitar a Proveedor) con totales + exportar. Botón **"Generar solicitud" deshabilitado ("próximamente")** = gancho de persistencia (sub-proyecto C).

## 11. Transversal

- **Rendimiento:** reusa `#refs` cacheado para acotar todo al proveedor; Siembra/Inventario/Hold son snapshots (consulta directa); Ventas con pushdown de fecha + `WITH (NOLOCK)` (patrón G00); catálogos cacheados (TTL diario). Cascada en memoria PHP (barata). `Bodegas` siempre con `AND CIA = 7` (gotcha fan-out de G00).
- **Errores:** 401 sin autenticar; 500 con detalle ante fallo de query; proveedor vacío (admin) → `#refs` vacío → todo en 0 sin errores; negocio sin siembra → faltante/sobrante = 0 (no rompe).
- **Pruebas:** motor con tests unitarios + fixtures (cubre solo reubicando / parcial CEDI / desborde a proveedor) — TDD. Endpoint con harness tipo G00 (sesión simulada + fingerprints) cuando existan las tablas.
